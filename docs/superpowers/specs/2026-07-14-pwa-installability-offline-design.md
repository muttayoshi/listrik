Status: Approved for planning

## Why this scope, why now

Four of the five P0 gaps are one dependency chain: the app cannot pass PRD ¬ß16's MVP
acceptance criteria ("installable + offline + lolos Lighthouse PWA") without a manifest,
a service worker, and an install affordance ‚Äî and none of those can be verified without
the other two existing first. Bundling them into one spec avoids re-deriving the same
build/config context three times.

The fifth item (PPJ% input) is small and independent, but the user chose to bundle it
into this same spec rather than run it separately.

## Context worth remembering while implementing

This project runs inside **Figma Make**, not a plain Vite scaffold:

- `vite.config.ts` sets `base` dynamically from `process.env.FIGMA_PUBLIC_URL`. Any PWA
  config that hardcodes `start_url`/`scope`/asset paths will break when this env var is
  set. Use relative values so Vite's `base` resolution applies automatically.
- `index.html` has literal comment markers `<!-- figma:head-start -->` /
  `<!-- figma:head-end -->` and `<!-- figma:body-start -->` / `<!-- figma:body-end -->`
  that the `figma-site-configuration` Vite plugin does a raw string-replace on. Do not
  place new static tags (iOS meta tags, manual manifest link, etc.) *between* those exact
  comment strings if it can be avoided ‚Äî add them elsewhere in `<head>` so the plugin's
  string replacement can't clobber them. `vite-plugin-pwa`'s own HTML injection runs via
  a separate `transformIndexHtml` hook and composes fine with the existing plugins.
- `AGENTS.md`: a dev server is *always* running (`vite --host 0.0.0.0`, port from
  `$PORT`). Service worker behavior must not interfere with HMR/dev ‚Äî keep
  `devOptions.enabled: false` so the SW only activates in production builds.
- No image-conversion tooling (ImageMagick, rsvg-convert, inkscape) and no
  canvas/sharp-style npm packages are available in this environment. Icons must be
  generated as raw PNG bytes using only Node's built-in `zlib` (manual PNG chunk
  encoding), or supplied by the user later.
- No Lighthouse CLI / Chrome available in this environment. Lighthouse's automated
  PWA score **cannot** be run here ‚Äî see Verification section.

## 1. Web App Manifest & icons

- Add `vite-plugin-pwa@^1.3.0` (peer-compatible with Vite 8) using the `generateSW`
  strategy (Workbox, zero manual SW code ‚Äî per PRD ¬ß10.2 and ¬ß13 recommendation).
- Manifest fields:
  - `name`: `"Listrikku ‚Äî Kalkulator Biaya Listrik"`
  - `short_name`: `"Listrikku"` (matches the existing Dexie DB name `'listrikku'` in
    `src/db.ts:57`)
  - `theme_color`: `"#059669"` (== `--color-brand-600` in `src/index.css`)
  - `background_color`: `"#f8fffe"` (== `--color-surface` in `src/index.css`)
  - `display`: `"standalone"`
  - `start_url`: `"."`, `scope`: `"."` ‚Äî relative, so they resolve against Vite's
    configured `base` (including the dynamic `FIGMA_PUBLIC_URL` case) instead of a
    hardcoded `/`.
  - `icons`: 192√ó192, 512√ó512, one `purpose: "maskable"` 512√ó512, all PNG.
- Icons are generated by a one-off Node script (uses only `zlib`, no new dependency)
  producing raw PNG files into `public/icons/`: emerald gradient
  (`#059669 ‚Üí #047857`, matching the header logo in `src/App.tsx:52-57`) with a simple
  ‚ö° glyph. The maskable variant pads the glyph inside the safe-zone circle (icon
  content within the center ~80%) per PWA maskable-icon requirements. An additional
  180√ó180 non-transparent PNG is generated for `apple-touch-icon`.
- `apple-mobile-web-app-capable` (and related `apple-mobile-web-app-*` meta tags) plus
  `<link rel="apple-touch-icon">` are added directly in `index.html`'s `<head>`, outside
  the `figma:head-start`/`figma:head-end` comment markers.
- These placeholder icons satisfy the manifest/installability criteria now; swapping in
  final branded icons later is a drop-in replacement of the same files, no code change.

## 2. Service worker & offline behavior

- `generateSW` strategy, `registerType: 'autoUpdate'` ‚Äî when a new SW version is ready,
  it activates and the page reloads automatically, silently. **This is a deliberate
  deviation from PRD ¬ß10.2**, which specifies a manual "Versi baru tersedia ‚Äî muat ulang"
  prompt. The user chose auto-update for simplicity; PRD ¬ß10.2 should be updated to
  reflect this if/when the PRD is revised for accuracy (see `TODO-PRD-GAPS.md`'s existing
  "PRD belum mencerminkan implementasi" pattern for rooms ‚Äî this is the same kind of
  drift, tracked here explicitly instead of silently).
- `globPatterns` cover the full production app shell (JS/CSS/HTML/icons) ‚Üí
  precache + cache-first. No runtime caching config needed ‚Äî the app makes no runtime
  network calls (PRD ¬ß10.2 confirms no API dependency).
- `devOptions.enabled: false` ‚Äî the SW is only registered against production builds
  (`vite build && vite preview`), never in `vite dev`, so it can't interfere with the
  always-on dev server's HMR.
- IndexedDB (Dexie) user data is completely independent of the Workbox SW cache ‚Äî
  clearing the SW cache never touches `rooms`/`devices`/`settings` tables (PRD ¬ß10.4).
  No code change needed here; this is a property of using separate browser storage
  systems, worth confirming during verification rather than implementing.

## 3. Installability ‚Äî install banner & iOS instructions

New component `src/components/InstallBanner.tsx`, mounted in `App.tsx` above `<main>`
(dismissible, non-blocking ‚Äî a thin banner, not a modal).

- **Android / Chrome desktop:** listen for the `beforeinstallprompt` event, capture and
  store the event, render a "Pasang aplikasi" button. On click, call the stored event's
  `.prompt()`.
- **iOS Safari:** detected via `navigator.userAgent` (iOS check) combined with
  `navigator.standalone === false`. `beforeinstallprompt` never fires here, so instead
  render static instruction text: "Bagikan ‚Üí Tambah ke Layar Utama" (no actionable
  button possible).
- Banner never renders if the app is already running standalone (checked via
  `window.matchMedia('(display-mode: standalone)').matches` or `navigator.standalone`).
- Dismissal is permanent: writes a flag to `localStorage` (e.g.
  `installBannerDismissed: "1"`), checked on mount so the banner doesn't reappear on
  future visits once dismissed.

## 4. PPJ% input (`SettingsPanel.tsx`)

- New number input, following the exact pattern already used for "Hari per Bulan"
  (`src/components/SettingsPanel.tsx:88-101`): local `useState` seeded from
  `settings.ppjPercent`, label `"PPJ (%)"`, validated range 0‚Äì10 inclusive, default `0`.
- Wired into the existing `handleSave` validation block and the `updateSettings` call
  alongside `tariffPerKwh` and `daysPerMonth`. No `db.ts` or `store.ts` changes needed ‚Äî
  `ppjPercent` already exists on the `Settings` type and is already consumed by
  `calcDevice` (`src/db.ts:84`).

## 5. Verification

- Build and serve production output: `vite build && vite preview`.
- Confirm via browser DevTools (Application tab): manifest loads and validates, all
  required manifest fields present, SW registers successfully, precache list includes
  the app shell files.
- Confirm offline behavior by simulating offline in DevTools (Network ‚Üí Offline) and
  reloading ‚Äî app shell must load and the full flow (open ‚Üí add room/device ‚Üí see
  totals) must work without a network connection, per PRD ¬ß10.4 / ¬ß16.
- **Limitation, stated explicitly:** this environment has no Chrome/Lighthouse CLI
  available, so an automated Lighthouse PWA score **cannot be produced as part of this
  work**. Verification here will instead produce a manual checklist covering each
  Lighthouse PWA criterion (installable, works offline, manifest valid, etc.). Before
  treating PRD ¬ß16 as fully satisfied, the user should run a real Lighthouse audit
  in an actual Chrome browser against the built app.
